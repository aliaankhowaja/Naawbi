## 3. Structured Specifications (Sprint 3)

### Spec 1: US 1–4 — Authentication

| Field | Detail |
|---|---|
| **Actor** | Any user (Student or Instructor) |
| **Inputs** | Email (required), password (required) |
| **Processing** | Hash submitted password with SHA-256; compare against stored hash; if match, load user record into Session singleton with id, username, email, and role |
| **Outputs** | User redirected to CourseCatalogView; session state available application-wide via `Session.getInstance()` |
| **Preconditions** | User account exists in the `users` table with a hashed password |
| **Postconditions** | Session populated; logout clears Session and returns to LoginView |
| **Exceptions** | Wrong credentials → inline error label "Incorrect email or password"; empty fields → validation error |

---

### Spec 2: US 5 — Role-Based Access Control

| Field | Detail |
|---|---|
| **Actor** | System (enforced on every protected action) |
| **Inputs** | Logged-in user role from `Session.getInstance().getRole()` |
| **Processing** | On CourseCatalogController initialisation: query `course_enrollments` for student; query `courses.created_by` for instructor; filter course list accordingly. On instructor-only action: check role before showing form — deny with an Access Denied dialog if role is `student` |
| **Outputs** | Students see only enrolled courses; Instructor-only buttons (Post Announcement, Create Assignment) hidden or guarded for students |
| **Preconditions** | User is logged in |
| **Postconditions** | No student can perform instructor actions; no instructor can see student-only submission details for other students |
| **Exceptions** | Unauthenticated access attempt → redirect to LoginView |

---

### Spec 3: US 12 & 13 — Announcement Attachments

| Field | Detail |
|---|---|
| **Actor** | Instructor |
| **Inputs** | Optional file via system FileChooser OR a typed URL string |
| **Processing** | If a file is selected: store file path and original filename in `announcement_attachments` with `is_link = false`. If a URL is provided: store URL in `link_url` with `is_link = true`. Both types linked to announcement by foreign key. |
| **Outputs** | Attachment chip rendered on the announcement card in the course stream; file chip opens file, link chip opens URL |
| **Preconditions** | Announcement has been created or is being created; instructor is logged in |
| **Postconditions** | Attachment record persisted; visible to all enrolled students |
| **Exceptions** | No attachment selected → announcement saves without attachment (attachments are optional) |

---

### Spec 4: US 15 & 16 — Assignments Tab and People Tab

| Field | Detail |
|---|---|
| **Actor** | Student and Instructor |
| **Inputs** | Current course selection |
| **Processing** | Assignments tab: call `Assignment.fetchWithStatusForUser(courseId, userId)` which LEFT JOINs submissions; derive display status (Submitted / Late / Missing / Not Submitted) per assignment. People tab: call `Course.fetchEnrolledUsers(courseId)`; separate into Instructor and Student sections. |
| **Outputs** | Assignments tab: list of assignment cards with point values, deadlines, late-policy badge, and status badge (students) or "Grade Submissions" button (instructors). People tab: roster cards with role badge, avatar initial, username, and email. |
| **Preconditions** | A course is selected in the catalog; user is enrolled or is the course instructor |
| **Postconditions** | Both tabs refresh on each course selection |
| **Exceptions** | No assignments → empty state label; no enrolled users → empty state label |

---

### Spec 5: US 19 & 20 — Instructor Grading Interface

| Field | Detail |
|---|---|
| **Actor** | Instructor |
| **Inputs** | Selected submission; numeric grade (required); feedback text (optional) |
| **Processing** | Load all submissions for the assignment via `Submission.fetchByAssignmentId(assignmentId)`; on Save: validate grade is numeric and within 0 – total_points; call `Submission.saveGrade(submissionId, grade, feedback)`; update DB record |
| **Outputs** | Grade and feedback persisted to `submissions.grade` and `submissions.feedback`; ListView item updates to show graded state |
| **Preconditions** | Instructor is logged in; at least one submission exists for the assignment |
| **Postconditions** | Submission record has `grade` and `feedback` values; visible in student gradebook |
| **Exceptions** | No submissions for assignment → empty state label in GradingView; invalid grade input → inline validation message |

---

### Spec 6: US 21 & 22 — Gradebook Views

| Field | Detail |
|---|---|
| **Actor** | Student (personal grades) and Instructor (class gradebook) |
| **Inputs** | Current course ID; logged-in user role |
| **Processing** | Student: call `Submission.fetchGradedForStudent(courseId, userId)`; display each graded assignment with score, out-of value, and feedback. Instructor: call `Submission.fetchGradebookForCourse(courseId)`; build a GridPane with enrolled students as rows and course assignments as columns; populate grade cells. |
| **Outputs** | Student sees a personal grade table with feedback. Instructor sees a scrollable class grid with numeric grades or "—" for ungraded submissions. |
| **Preconditions** | Course is selected; user is enrolled (student) or owns the course (instructor) |
| **Postconditions** | Read-only display; no mutations from Gradebook views |
| **Exceptions** | No graded submissions → empty state message "No grades yet." |

# PART C — Deliverable 3 Document

## 1. Introduction

### Context

Naawbi is a university-level Learning Management System built iteratively by Team Ibwaan for CS3009 Software Engineering (Spring 2026). Deliverable 3 represents the final sprint of the project and closes the remaining feature gaps: user authentication, role enforcement, announcement enrichment, and the complete grading pipeline.

### Progress from Sprint 2 to Sprint 3

Sprint 2 gave the platform its content layer — announcements, assignments, file submissions, and a student to-do list. All of that was accessible without any authentication. Sprint 3 locks it down: every user now logs in with a verified identity, sees only what their role permits, and participates in a full grade cycle from submission through instructor evaluation to personal gradebook.

### Problem Statement

The problem remains unchanged: fragmented academic tooling creates friction for both instructors and students. Sprint 3 completes the platform's value proposition — students can now see their grades alongside their submissions in one place, and instructors have a single interface from which to post content, grade work, and review class performance.

---

## 2. Project Vision

Build a clean, reliable, course-centric LMS that instructors use to manage content and evaluations, and that students use to track progress and performance — without requiring external tools for any core academic workflow.

Sprint 1 built the course structure. Sprint 2 populated it with content and submissions. Sprint 3 closes the loop: authentication, role enforcement, grading, and gradebook. The full feature set is now delivered.

## 6. Non-Functional Requirements (NFR) — Final State

| NFR Category | Requirement |
|---|---|
| **Security** | Passwords are never stored or compared in plain text — SHA-256 hashing is applied at all registration and login points. Session state is held in a singleton cleared on logout; no credentials persist beyond the session. RBAC guards are applied at the controller level on every instructor-only action. |
| **Performance** | Gradebook queries (class-wide grid) are designed to execute in a single JOIN rather than N+1 queries. All primary views load within 2–3 seconds on typical lab hardware at class sizes up to ~200 students per course. |
| **Reliability** | Grade saves are atomic: `Submission.saveGrade()` issues a single parameterised UPDATE. No partial grade state is possible. Session logout is synchronous before the LoginView is displayed. |
| **Usability** | Role-adaptive UI: instructor-only buttons are hidden (not just disabled) for students. Gradebook cells use "—" for ungraded rather than nulls. Status badges (Submitted / Late / Missing / Not Submitted) use both colour and text, never colour alone. |
| **Maintainability** | All Sprint 3 features follow the existing layered MVC architecture (controller / model / view). Business logic (RBAC checks, grade validation, gradebook query) is in model/controller classes, not in FXML event handlers. |
| **Auditability** | All graded submissions retain original submission timestamp alongside the grade and feedback. Session login events are gated on credential verification; no anonymous access path remains in the application. |

## 9. Retrospective

### What went well
- Feature delivery was on schedule; all eight Sprint 3 user stories were completed before the submission deadline.
- The layered MVC architecture established in Sprint 1 paid off — adding the grading pipeline required no restructuring of existing code.
- The seeded test database (`seed.sql`) with pre-configured accounts and realistic submission states significantly accelerated manual testing and team handoff.
- Branch-per-feature strategy with descriptive PR titles made code review straightforward; no rework was needed after merges.

### What could be improved
- Authentication was implemented later than ideal — earlier sprints operated without a session guard. Future projects should implement auth in Sprint 1.
- The gradebook-db / grading-ui / gradebook branch chain created a sequential merge dependency; parallel work was limited during that period.

### What we would do differently
- Implement user registration and login in Sprint 1 so all subsequent features build on authenticated identity from the start.
- Use branch targeting (feature → integration → main) instead of all-to-main to allow parallel grading feature work with less merge risk.
